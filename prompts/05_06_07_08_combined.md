## Contexto
O materiais.py atual funciona razoavelmente — listagem, detalhe, toggle ai_enabled, sync com Canvas. Mas o toggle é cosmético (não gera/remove embeddings), falta upload manual, cooldown é 1h em vez de 4h, e falta size_bytes/size_label no response.

## Pré-requisitos
Prompts 00 e 01 executados.

## Arquivos a MODIFICAR
- `coffee-backend/app/routers/materiais.py`
- `coffee-backend/app/schemas/materiais.py`

## Tarefa

### 1. Adaptar `schemas/materiais.py`
Adicionar ao MaterialResponse: `size_bytes: Optional[int]`, `size_label: Optional[str]`.
Criar `MaterialUploadRequest` (não é schema — é multipart, mas documentar).

### 2. Adaptar `routers/materiais.py`

**Mudanças no PATCH /toggle-ai:**
Após inverter o flag, chamar:
- Se ai_enabled mudou pra True: chamar `generate_material_embeddings(texto_extraido, material_id, disciplina_id)` do embedding_service
- Se ai_enabled mudou pra False: chamar `remove_embeddings(material_id)` do embedding_service
- Buscar texto_extraido do material pra passar pro embedding service

**Adicionar POST /disciplinas/{id}/materiais (upload manual):**
- multipart/form-data: file (max 20MB), ai_enabled (default true)
- Extrair texto do arquivo (reuse lógica do scraper: PyPDF2, python-pptx)
- Upload pro Supabase Storage (reuse lógica do scraper/storage.py adaptada)
- Salvar no banco com fonte='manual'
- Se ai_enabled=true, gerar embeddings
- Retorna material criado

**Mudar cooldown:** de 1h pra 4h no trigger_sync.

**Adicionar size_label:** nos responses de listagem e detalhe, computar size_label a partir de size_bytes.

### 3. Envelope padrão em todos os responses.

## Verificação
1. PATCH /toggle-ai gera embeddings quando ai_enabled vai pra true
2. PATCH /toggle-ai remove embeddings quando ai_enabled vai pra false
3. POST /disciplinas/{id}/materiais faz upload e extrai texto
4. Cooldown é 4 horas (não 1 hora)
5. Responses incluem size_bytes e size_label

# Prompt 06 — Auth (Expandir)

## Contexto
O auth.py atual tem signup, login, e /me. Faltam: referral_code no signup, logout, forgot-password, refresh, e dados expandidos no response (plano, trial_end, espm_connected).

## Pré-requisitos
Prompt 00 executado (tabela referrals existe, users tem plano/trial_end/referral_code).

## Arquivos a MODIFICAR
- `coffee-backend/app/routers/auth.py`
- `coffee-backend/app/schemas/auth.py`

## Tarefa

### 1. Reescrever `schemas/auth.py`
```python
class SignupRequest(BaseModel):
    nome: str = Field(min_length=2, max_length=255)
    email: EmailStr
    password: str = Field(min_length=8, max_length=128)
    referral_code: Optional[str] = None

class LoginRequest(BaseModel):
    email: EmailStr
    password: str

class LogoutRequest(BaseModel):
    device_token: Optional[str] = None

class ForgotPasswordRequest(BaseModel):
    email: EmailStr

class UserResponse(BaseModel):
    id: UUID
    nome: str
    email: str
    plano: str
    trial_end: Optional[datetime]
    subscription_active: bool
    espm_connected: bool
    referral_code: Optional[str]
    created_at: datetime

class AuthResponse(BaseModel):
    user: UserResponse
    token: str

class TokenResponse(BaseModel):
    token: str
```

### 2. Adaptar `routers/auth.py`

**POST /signup** — Expandir:
- Gerar referral_code único ao criar user (formato: primeiras 5 letras do nome uppercase + ano, ex: GABRI2026. Se conflito, adicionar random suffix)
- Setar plano='trial', trial_end=NOW()+7 dias
- Se referral_code fornecido: validar que existe, criar registro em referrals, adicionar +7 dias ao trial_end do referrer
- Criar registro em user_settings com defaults
- Response retorna UserResponse expandido

**POST /login** — Expandir response:
- Query expandida: buscar plano, trial_end, referral_code
- Computar subscription_active: true se plano='premium' e exists subscription ativa
- Computar espm_connected: true se espm_login IS NOT NULL
- Response retorna UserResponse expandido

**POST /logout** — NOVO:
- Recebe: device_token (opcional)
- Se device_token fornecido: DELETE de device_tokens onde fcm_token = device_token
- Se não fornecido: DELETE todos device_tokens do user
- Retorna { data: null, message: "Logged out" }

**POST /forgot-password** — NOVO:
- Recebe: email
- Sempre retorna 200 (segurança — não revela se email existe)
- TODO: integrar com Supabase Auth password reset ou sistema de email

**POST /refresh** — NOVO:
- Pega token atual do header Authorization
- Gera novo JWT
- Retorna { data: { token: "novo..." } }
- Se token expirado: 401 TOKEN_EXPIRED

**GET /me** — Adaptar pra retornar UserResponse expandido (mesmos campos do login).

## Verificação
1. POST /signup gera referral_code automaticamente
2. POST /signup aceita referral_code e aplica +7 dias ao referrer
3. POST /login retorna plano, trial_end, espm_connected, subscription_active
4. POST /logout remove device tokens
5. POST /forgot-password sempre retorna 200
6. POST /refresh gera novo JWT
7. GET /me retorna dados completos
8. Envelope padrão

# Prompt 07 — Profile, Subscription, Referral, Settings (Novos)

## Contexto
4 novos routers que não existem no backend atual.

## Pré-requisitos
Prompts 00 e 06 executados.

## Arquivos a CRIAR
- `coffee-backend/app/routers/profile.py`
- `coffee-backend/app/routers/subscription.py`
- `coffee-backend/app/routers/referral.py`
- `coffee-backend/app/routers/settings.py`
- `coffee-backend/app/schemas/profile.py`
- `coffee-backend/app/schemas/subscription.py`
- `coffee-backend/app/schemas/referral.py`
- `coffee-backend/app/schemas/settings.py`

## Arquivos a MODIFICAR
- `coffee-backend/app/main.py` — registrar 4 novos routers

## Tarefa

### GET /profile
Retorna dados completos do usuário:
- Dados pessoais (nome, email, plano, trial_end, espm_connected, referral_code)
- referrals_count: COUNT de referrals WHERE referrer_id = user_id
- usage.gravacoes_total: COUNT de gravacoes WHERE user_id
- usage.horas_gravadas: SUM(duration_seconds) / 3600.0 das gravações
- usage.perguntas_hoje: COUNT de mensagens WHERE role='user' AND chat.user_id AND created_at >= hoje
- usage.perguntas_limite: 10 se trial, -1 se premium

### PATCH /profile
Atualiza nome. Retorna profile atualizado.

### POST /subscription/verify
Recebe receipt_data e transaction_id. Por enquanto: validação básica (salvar na tabela subscriptions, setar plano='premium'). Integração completa com Apple Server-to-Server virá depois.

### GET /subscription/status
Retorna plano, subscription_active, expires_at.

### GET /referral
Retorna code, total_referrals, days_earned (total_referrals * 7), share_message (pré-formatada), lista de referrals com nome e data.

### POST /referral/validate
Recebe code. Retorna { valid: true/false, referrer_name: "..." }.

### GET /settings
Busca user_settings (ou cria com defaults se não existe). Retorna todos os campos.

### PATCH /settings
Atualiza apenas campos enviados (partial update). Retorna settings atualizados.

## Verificação
1. GET /profile retorna usage computado corretamente
2. POST /subscription/verify salva na tabela e atualiza plano
3. GET /referral lista referrals com nomes
4. POST /referral/validate retorna valid e referrer_name
5. GET /settings cria defaults na primeira chamada
6. PATCH /settings faz partial update
7. Todos registrados no main.py
8. Envelope padrão
